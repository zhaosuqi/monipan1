# K线批量获取优化说明

## 优化内容

根据用户需求"补充1m k线，要从最后一条一直到不到当前时刻，如果一次需要补多条，要合理计算每次获取数量多补几次"，优化了 `fetch_klines_to_1m.py` 的获取逻辑。

## 优化前的问题

### 旧逻辑

```python
# 计算需要获取的K线数量
duration_minutes = int((end_time - start_time).total_seconds() / 60)

# 限制每次最多获取1000条
limit = min(duration_minutes + 1, 1000)

# 获取K线数据
klines = exchange.get_klines(limit=limit)

# 只获取一次，保存后退出
```

**问题：**
- 如果需要补充10000条K线，只会获取1000条
- 需要等待下一个循环（10秒后）再获取
- 效率低下，补充大量历史数据时速度慢

## 优化后的逻辑

### 新逻辑（分批次获取）

```python
# 计算需要补充的总分钟数
total_minutes = int((end_time - last_time).total_seconds() / 60)

# 分批次获取，每次最多1000条
batch_count = 0
while last_time < end_time:
    batch_count += 1

    # 每次固定获取1000条（币安API最大值）
    new_klines = fetch_new_klines(exchange, last_time, end_time)

    # 保存到数据库
    saved_count = save_to_klines_1m(new_klines)

    if saved_count > 0:
        # 更新最后时间
        last_time = new_klines[-1].close_time

        # 如果获取数量少于1000，说明已经补完
        if saved_count < 1000:
            break

    # 避免请求过快
    time.sleep(0.5)
```

**优势：**
- 在一个循环周期内分批次获取所有数据
- 每批最多1000条，避免API限制
- 获取数量 < 1000 时自动停止，不浪费请求
- 批次间间隔0.5秒，避免请求过快

## 关键修改

### 1. fetch_new_klines() 函数

**修改前：**
```python
# 计算需要获取的K线数量
duration_minutes = int((end_time - start_time).total_seconds() / 60)
limit = min(duration_minutes + 1, 1000)

klines = exchange.get_klines(limit=limit)
```

**修改后：**
```python
# 每次固定获取1000条（币安API的最大值）
limit = 1000

klines = exchange.get_klines(limit=limit)

# 按时间升序排列
finished_klines.sort(key=lambda x: x.close_time)
```

**改进：**
- 简化逻辑，固定使用1000条
- 返回的K线按时间升序排列
- 过滤出符合时间范围的K线

### 2. 主循环 - 批量获取逻辑

**新增：**
```python
# 计算需要补充的总分钟数
total_minutes = int((end_time - last_time).total_seconds() / 60)
logger.info(f"需要补充约 {total_minutes} 条K线")

# 分批次获取
batch_count = 0
while last_time < end_time:
    batch_count += 1
    logger.info(f"→ 批次 {batch_count}: 从 {last_time} 开始获取")

    new_klines = fetch_new_klines(exchange, last_time, end_time)

    if not new_klines:
        break

    saved_count = save_to_klines_1m(new_klines)

    if saved_count > 0:
        total_fetched += saved_count
        last_time = new_klines[-1].close_time

        logger.info(
            f"✓ 批次 {batch_count} 完成，"
            f"本批获取 {saved_count} 条，"
            f"累计获取 {total_fetched} 条K线"
        )

        # 如果获取数量少于1000，说明已经补完
        if saved_count < 1000:
            logger.info(
                f"本批获取数量 {saved_count} < 1000，"
                f"说明已补完所有数据"
            )
            break

    # 避免请求过快
    if last_time < end_time:
        time.sleep(0.5)
```

## 关键参数

### 批次间隔

```python
# 避免请求过快
if last_time < end_time:
    time.sleep(0.5)  # 每批间隔0.5秒
```

**可调整：**
- 币安API限制：每分钟1200次请求
- 0.5秒间隔 = 每秒2次 = 每分钟120次
- 远低于API限制，安全

### 单批数量

```python
limit = 1000  # 币安API的最大值
```

**固定值：**
- 币安API的limit参数最大值为1000
- 不能超过这个值

### 停止条件

```python
if saved_count < 1000:
    logger.info("本批获取数量 < 1000，说明已补完所有数据")
    break
```

**逻辑：**
- 如果获取数量 < 1000，说明已经补完
- 避免多余的API请求

## 使用建议

### 日常运行（少量新数据）

**预期：**

- 在一个批次内完成
- 快速高效

**场景：** 补充到一半时程序中断

**恢复：**
```bash

./start_fetch_klines_to_1m.sh
```

**逻辑：**
- 从数据库中最后一条K线时间继续
- 自动补充剩余数据
- 不会重复插入（INSERT OR REPLACE）

## 注意事项

1. **API限制**
   - 币安API每分钟限制1200次请求
   - 当前间隔0.5秒，每分钟约120次，安全
   - 不建议再缩短间隔

2. **网络稳定性**
   - 批量获取需要稳定的网络连接
   - 如果网络不稳定，可能需要多次重试

3. **数据库写入**
   - 使用 INSERT OR REPLACE，避免重复
   - 每批提交一次事务
   - 确保数据一致性

4. **内存占用**
   - 每批处理1000条，内存占用很小
   - 可以安全运行

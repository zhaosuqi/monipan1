## 工作原理

### 1. 智能时间控制

```python
def wait_for_next_query_time():
    """等待到下一个查询时间点"""
    # 在每分钟的 01, 11, 21, 31, 41, 51 秒执行
    # 避开整点，确保K线已完结
```

**为什么选择这些时间点？**
- ❌ **00秒** - K线可能还未形成
- ✅ **01秒** - 足够延迟，确保上一分钟K线已完结
- ✅ **10秒间隔** - 分散负载，避免API限流
- ✅ **最多10秒延迟** - 保证实时性

### 2. 只获取已完结的K线

```python
# 计算本次查询的结束时间（上一分钟，确保K线已完结）
end_time = now.replace(second=0, microsecond=0) - timedelta(minutes=1)

# 过滤：只保留已完结的K线（close_time在当前时间之前）
if kline.close_time < now:
    finished_klines.append(kline)
```

**示例：**
- 现在是 `10:03:45`
- 查询到 `10:02:00` 的K线（已完结）
- **不包含** `10:03:00` 的K线（正在形成中）

### 3. 增量更新机制

```python
# 从数据库最后时间开始
last_time = klines_1m最后一条时间  # 例如: 2025-12-24 07:05:59
```

## 注意事项

### 1. 初次运行时间

首次运行时，如果数据库最后时间离现在较远（如测试数据显示23217分钟），会一次性获取大量历史数据。

**建议：**
- 初次运行时观察几分钟
- 确认数据正常补充后再放心使用
- 可以先用测试脚本验证

### 2. 指标计算

指标计算需要足够的历史数据：
- MACD: 至少26条数据
- 长期MA: 至少200天数据

如果数据不足，某些指标可能为NULL。

### 3. 网络稳定性

- 使用可靠的网络环境
- API调用失败会自动重试
- 长时间断网会导致数据缺口

### 4. 时间同步

确保系统时间准确：
```bash
# macOS
sudo sntp -sS time.apple.com

# Linux
sudo ntpdate pool.ntp.org
```

## 故障排查

### 问题: 没有获取到新K线

**检查：**
```bash
# 1. 查看当前时间是否已最新
sqlite3 data/klines.db \
  "SELECT close_time FROM klines_1m ORDER BY close_time DESC LIMIT 1"

# 2. 查看当前时间
date

# 3. 如果差距在1分钟内，说明数据已最新
```

### 问题: 程序运行一段时间后停止

**原因：**
- 网络中断
- API限流
- 程序异常

**解决：**
```bash
# 使用systemd或supervisor守护进程
# 添加自动重启机制
nohup ./start_fetch_klines.sh > logs/fetch.log 2>&1 &
```

### 问题: 指标列为NULL

**原因：** 历史数据不足

**解决：** 等待更多数据积累，某些长期指标需要200天数据。

# 回测开始日期修复 - 时间格式问题

## 日期
2026-01-07

## 用户问题

"为什么回测的数据第一条是2024年1月2日 0点？"
"配置是2024-01-01，但实际从2024-01-02开始"

## 问题根源

### 时间格式不匹配

**数据库存储格式**: `2024-01-01 00:00:00+00:00`（使用空格）

**代码查询格式**: `2024-01-01T00:00:00`（使用T）

**字符串比较**:
```
'2024-01-01 00:00:00+00:00' < '2024-01-01T00:00:00'
```

因为空格(` `)的ASCII码(32)小于`T`的ASCII码(84)，所以：
- `2024-01-01 00:00:00` < `2024-01-01T00:00:00`
- 查询 `>= '2024-01-01T00:00:00'` 会跳过所有 `2024-01-01 ` 的数据！

### 文件修改

**文件**: [data_module/db_kline_reader.py:134-136](data_module/db_kline_reader.py#L134-L136)

```python
start_time_normalized = start_time.replace(' ', 'T')
```


**修复**: 将T转换为空格，匹配数据库存储格式

## 技术细节

### SQLite字符串比较

SQLite使用ASCII码进行字符串比较：

```sql
-- ASCII码
-- 空格(' ') = 32
-- 'T' = 84

-- 因此
'2024-01-01 00:00:00' < '2024-01-01T00:00:00'  -- True
'2024-01-01 00:00:00' >= '2024-01-01T00:00:00' -- False
```

### 时间格式示例

| 格式 | 示例 | 说明 |
|------|------|------|
| **数据库格式** | `2024-01-01 00:00:00+00:00` | 使用空格 |
| **ISO格式** | `2024-01-01T00:00:00+00:00` | 使用T |
| **配置格式** | `2024-01-01 00:00:00` | 使用空格 |

## 相关问题

### Q: 为什么数据库存储格式有空格？

A: SQLite存储文本时保留了原始格式。数据导入时可能是这种格式：
```python
datetime.strftime('%Y-%m-%d %H:%M:%S%z')  # 产生 "2024-01-01 00:00:00+00:00"
```

### Q: 为什么原来要转换成T？

A: 原代码认为T是标准ISO格式，但没有考虑数据库实际存储格式。

### Q: 现在兼容两种格式吗？

A: 是的！修改后的代码会：
- 将T转换为空格
- 保持空格不变
- 所以无论输入是 `2024-01-01 00:00:00` 还是 `2024-01-01T00:00:00`，都能正确查询
